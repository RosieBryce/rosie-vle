Introduction to Neural Networks and PyTorch

PyTorch is the research-first framework — more explicit than Keras, more Pythonic than TF1. The learning curve is steeper but the control is worth it. The training loop is manual here, which forces you to actually understand what's happening.

Autograd

How PyTorch Autograd Works

PyTorch builds a dynamic computation graph as operations execute. When you call .backward(), it traverses this graph in reverse to compute gradients (the chain rule, applied automatically).

ConceptWhat it means
requires_grad=TrueMarks a tensor: "track operations on me for gradient computation"
tensor.backward()Computes ∂loss/∂w for all tensors with requires_grad=True; results stored in .grad
tensor.gradAccumulated gradient — not reset automatically between batches
torch.no_grad()Context manager: disables graph building; use during evaluation to save memory
tensor.detach()Removes one tensor from the graph; the rest of the graph still exists

Key design choice: Gradients accumulate — they add to .grad on each .backward() call without being reset. You must call optimizer.zero_grad() to clear them before the next batch.

Training Loop

The 6-Step PyTorch Training Loop

Every PyTorch training loop follows the same pattern. Each step exists for a reason:

1
model.train() — enable dropout and BatchNorm batch-stats mode for the training epoch
2
optimizer.zero_grad() — clear accumulated gradients from the previous batch
3
outputs = model(X_batch) — forward pass; builds the computation graph
4
loss = criterion(outputs, y_batch) — compute scalar loss
5
loss.backward() — backprop: compute ∂loss/∂w for all parameters
6
optimizer.step() — update weights using the computed gradients
Zero_grad must come before forward, not after backward. If you call it after optimizer.step(), gradients from the current batch persist until the next batch starts, contaminating the next batch's gradients. The safe pattern: zero → forward → backward → step.
Loss Functions

CrossEntropyLoss and the Double-Softmax Trap

nn.CrossEntropyLoss is LogSoftmax + NLLLoss combined. It expects raw logits (unnormalised scores) from your model — it applies softmax internally.

TaskLoss functionModel's last layer
Multi-class classification nn.CrossEntropyLoss Raw logits (no softmax!)
Binary classification nn.BCEWithLogitsLoss Raw logits (no sigmoid!)
Regression nn.MSELoss Linear output (no activation)
If you add nn.Softmax to your model output AND use CrossEntropyLoss, you're applying softmax twice. The loss function receives already-squashed probabilities, then squashes them again — gradients become numerically wrong and training fails silently.

Exam Traps
⚠ Trap 1 — Forgetting zero_grad()
Gradients accumulate by default in PyTorch. If you don't call optimizer.zero_grad() at the start of each batch, gradients from all previous batches pile up. This is the #1 PyTorch mistake. The gradient each step sees is the sum of every gradient computed so far — training becomes incorrect.
⚠ Trap 2 — CrossEntropyLoss with Softmax output
nn.CrossEntropyLoss applies softmax internally. If your model also applies softmax, you double-apply it. The result is a probability distribution passed through softmax again — the distribution becomes near-uniform for large values, and gradients vanish. Model output should be raw logits.
⚠ Trap 3 — model.eval() vs model.train()
Forgetting to switch between train and eval modes causes incorrect evaluation scores. model.train() enables dropout (randomly zeroes neurons) and uses per-batch BatchNorm stats. model.eval() disables dropout and uses stored running statistics. Evaluating with model.train() active means dropout randomly degrades your results.
⚠ Trap 4 — loss without .item()
Accumulating total_loss += loss keeps the entire computation graph alive. Each loss tensor holds a reference to the graph that produced it. Accumulating tensors directly causes a memory leak that grows with each batch. Use total_loss += loss.item() to extract the Python float and release the graph.
⚠ Trap 5 — view vs reshape
view requires contiguous memory; reshape always works. After operations like permute or transpose, a tensor's memory may not be contiguous. Calling .view() on it raises a RuntimeError. .reshape() copies if necessary — use it when you're unsure about contiguity.

Common Gotchas
  • Forgetting optimizer.zero_grad() — gradient accumulation, #1 PyTorch mistake
  • Using model.train() during evaluation — dropout randomly degrades scores
  • Not calling .item() when accumulating loss — memory leak from retaining computation graphs
  • Double softmax — model output + CrossEntropyLoss
  • Forgetting .to(device) on data tensors — device mismatch error
  • Integer vs float tensors — loss functions expect dtype=torch.float for inputs, torch.long for class labels
  • nn.ModuleList vs Python list — only ModuleList registers parameters in model.parameters()
Full Training Loop
def train_one_epoch(model, loader, optimizer, criterion, device):
    model.train()                            # 1. training mode
    total_loss = 0.0
    for X_batch, y_batch in loader:
        X_batch, y_batch = X_batch.to(device), y_batch.to(device)
        optimizer.zero_grad()                # 2. clear gradients
        outputs = model(X_batch)             # 3. forward pass
        loss    = criterion(outputs, y_batch)# 4. compute loss
        loss.backward()                      # 5. backprop
        optimizer.step()                     # 6. update weights
        total_loss += loss.item()            # .item() to avoid memory leak
    return total_loss / len(loader)

def evaluate(model, loader, criterion, device):
    model.eval()
    total_loss = 0.0
    with torch.no_grad():
        for X_batch, y_batch in loader:
            X_batch, y_batch = X_batch.to(device), y_batch.to(device)
            outputs = model(X_batch)
            total_loss += criterion(outputs, y_batch).item()
    return total_loss / len(loader)
nn.Module Pattern
class MLP(nn.Module):
    def __init__(self, input_dim, hidden_dim, output_dim):
        super().__init__()              # ALWAYS call super().__init__() first
        self.fc1 = nn.Linear(input_dim, hidden_dim)
        self.fc2 = nn.Linear(hidden_dim, output_dim)
        self.relu    = nn.ReLU()
        self.dropout = nn.Dropout(0.3)

    def forward(self, x):               # defines the computation
        x = self.relu(self.fc1(x))
        x = self.dropout(x)
        return self.fc2(x)              # raw logits — no softmax

model = MLP(784, 256, 10).to(device)
