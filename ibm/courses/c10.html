LoRA

LoRA — Low-Rank Adaptation

Instead of updating the full weight matrix W, LoRA learns two small matrices A and B whose product approximates the update:

W_new = W_pretrained + ΔW = W_pretrained + B × A
W: (d_out × d_in) — FROZEN
A: (r × d_in) — trainable, initialised randomly
B: (d_out × r) — trainable, initialised to ZERO
HyperparameterEffectTypical values
r (rank)Higher r = more parameters, more capacity, higher overfitting risk4, 8, 16, 64
lora_alphaScaling factor; effective LR = alpha/r; typically set to 2×r8, 16, 32
target_modulesWhich weight matrices to apply LoRA to (Q, V projections most common)q_proj, v_proj
B is initialised to zero so the LoRA contribution starts at zero. The model begins fine-tuning identical to the pretrained model and gradually learns the adaptation. At inference, B×A can be merged back into W — no added latency.
Instruction Tuning

Instruction Fine-Tuning — Format is Critical

Instruction tuning converts a base LM into an assistant by training on (instruction, response) pairs. The data format must match exactly between training and inference.

FormatUsed byKey feature
Alpaca Early open-source models instruction / input / output JSON fields
ChatML OpenAI, many others <|system|>, <|user|>, <|assistant|> role tags
Llama 3 Meta LLaMA 3 <|begin_of_text|>, <|start_header_id|> tags
Compute loss on response tokens only. Set prompt token labels to −100 (ignored by cross-entropy loss). Training on the instruction tokens wastes compute and confuses the model about what it should be generating.
Forgetting

Catastrophic Forgetting — Why It Happens and How to Mitigate It

Fine-tuning on a narrow task can cause the model to "forget" its general capabilities — weights shift to serve the new task at the expense of previously learned knowledge.

MitigationHow it helps
LoRA / PEFTMost parameters frozen — forgetting is minimal; only adapters change
Small learning rateSmall updates preserve most pretrained knowledge
Multi-task trainingMix general data with task-specific data
EWCElastic Weight Consolidation — penalises changes to weights important for previous tasks

Exam Traps
⚠ Trap 1 — LoRA adds no parameters at inference
After fine-tuning, LoRA matrices A and B can be merged back into W. The merged model has the same architecture and parameter count as the original. There is no inference overhead when the adapter is merged. model = model.merge_and_unload() before deployment.
⚠ Trap 2 — Instruction tuning ≠ RLHF
Instruction tuning (SFT) is supervised learning on (instruction, response) pairs. RLHF involves a separate reward model and reinforcement learning (PPO). SFT teaches the model to follow instructions; RLHF teaches it to follow human preferences. These are distinct stages — SFT always comes before RLHF.
⚠ Trap 3 — Chat template mismatch
Fine-tuning with one chat format and inferring with another is a common silent failure. The model learns to expect specific role tokens. If you fine-tune with Alpaca format but prompt with ChatML tokens, the model sees unfamiliar structure and produces degraded, confused output. Always apply the same template used during training.
⚠ Trap 4 — lora_alpha is not a learning rate
lora_alpha is a scaling factor, NOT the learning rate. The LoRA output is scaled by alpha/r before being added to the frozen weight computation. Higher alpha = more influence from LoRA adapters. The learning rate is a separate parameter set in the TrainingArguments / optimiser.
⚠ Trap 5 — Base model still must fit in memory
PEFT reduces training memory but the base model must still load. For a 7B model at fp16, that's ~14GB just for model weights — before gradients or optimiser state. PEFT reduces the latter two significantly but does not compress the base weights. QLoRA (Course 11) solves this by loading the base in 4-bit.

Common Gotchas
  • Chat template mismatch at inference — train and infer with the same format; use tokenizer.apply_chat_template()
  • Loss on prompt tokens — set prompt token labels to −100 to compute loss on response only
  • LoRA target modules vary by architecture — LLaMA: q_proj/v_proj; BERT: query/value; inspect model.named_modules()
  • Not saving adapter separately — model.save_pretrained() with PEFT saves only the adapter (MB, not GB)
  • Overfitting on small datasets — fine-tuning <1000 examples without regularisation often overfits; use dropout, weight_decay, early stopping
LoRA Pattern
from peft import LoraConfig, get_peft_model, TaskType

config = LoraConfig(
    task_type=TaskType.CAUSAL_LM,
    r=16,                          # rank
    lora_alpha=32,                 # scaling (typically 2*r)
    lora_dropout=0.1,
    target_modules=['q_proj', 'v_proj']  # LLaMA attention layers
)

model = get_peft_model(base_model, config)
model.print_trainable_parameters()
# trainable params: 4,194,304 || all params: 6,742,609,920 || 0.0622%

# Save adapter only (not full model)
model.save_pretrained('lora_adapter/')

# Merge for deployment (no inference overhead)
merged_model = model.merge_and_unload()
HuggingFace Trainer Pattern
from transformers import TrainingArguments, Trainer

training_args = TrainingArguments(
    output_dir='./results',
    num_train_epochs=3,
    per_device_train_batch_size=16,
    learning_rate=2e-5,           # small LR for fine-tuning
    weight_decay=0.01,
    evaluation_strategy='epoch',
    load_best_model_at_end=True,
)

trainer = Trainer(
    model=model,
    args=training_args,
    train_dataset=tokenised_train,
    eval_dataset=tokenised_val,
    compute_metrics=compute_metrics
)
trainer.train()
