Attention

Scaled Dot-Product Attention

Attention allows each token to look at every other token in the sequence and weight them by relevance. The formula:

Attention(Q, K, V) = softmax(QKᵀ / √d_k) · V
ComponentRoleIntuition
Q (Query)What am I looking for?The token asking a question
K (Key)What does each token offer?What each token "advertises"
V (Value)What information do I take?The actual content retrieved
√d_k scalingPrevent dot product magnitude growthLarge d_k → large dot products → softmax saturates → vanishing gradients
Attention is O(n²) in time and memory with respect to sequence length. A sequence of 1000 tokens requires a 1000×1000 attention matrix. This is why long context windows are expensive.
Transformer

Multi-Head Attention & The Transformer Block

Multi-head attention runs attention h times in parallel, each with different learned Q/K/V projections. Each head can focus on different aspects (syntax, coreference, semantics). With d_model=512 and 8 heads, each head has d_k = 64 (512/8).

Input tokens
  ↓ Token Embedding + Positional Encoding
  ↓ Multi-Head Self-Attention
  ↓ Add & Norm (residual + LayerNorm)
  ↓ Feed-Forward Network (Linear → ReLU → Linear)
  ↓ Add & Norm
  ↓ Output

Why residual connections? Gradients flow directly from output to input — prevents vanishing gradients in deep models (12–96 layers). LayerNorm vs BatchNorm: LayerNorm normalises per sample across features — better for variable-length sequences.

Positional Encoding

Positional Encoding — How the Model Knows Word Order

Transformers process all tokens in parallel — they have no inherent sense of order. Positional encoding injects position information.

MethodUsed byProperties
Sinusoidal (fixed) Original Transformer Not learned; allows generalising to longer sequences than seen in training
Learned (lookup table) BERT, GPT Trained with the model; limited to max_position_embeddings
RoPE (Rotary) LLaMA, Mistral Encodes relative positions via rotation; better extrapolation to longer sequences
Positional encoding is additive — added to the token embedding, NOT concatenated.

Exam Traps
⚠ Trap 1 — Self-attention is O(n²)
Attention complexity scales quadratically with sequence length. Doubling the sequence length quadruples the attention computation and memory. This is the core reason why efficient attention variants (FlashAttention, sparse attention) were developed.
⚠ Trap 2 — [CLS] token is always first
In BERT, [CLS] is prepended to the input — it is always token position 0. Its output embedding is used for sentence-level classification tasks. [SEP] separates sentences. The model uses token_type_ids (0 or 1) to distinguish which sentence a token belongs to.
⚠ Trap 3 — attention_mask is not optional when batching
When batching sequences of different lengths, padding tokens must be masked. Attention mask: 1 = attend, 0 = ignore. Without it, the model attends to padding tokens, which produces nonsensical context mixing and degrades performance. The intuition is counterintuitive: 1 means "pay attention", 0 means "mask out".
⚠ Trap 4 — RoBERTa removes NSP
RoBERTa improves on BERT by removing Next Sentence Prediction (NSP). NSP turned out to be unhelpful or even harmful to performance. RoBERTa also uses dynamic masking, larger batches, and more data. It is NOT architecturally different — same encoder-only transformer.
⚠ Trap 5 — Multi-head dimension arithmetic
With d_model=512 and 8 heads, each head has d_k = 64 (512/8). The total computation is the same as a single attention head with d_model=512. Multi-head attention does not increase parameters proportional to heads — the final projection keeps the output size constant.

Common Gotchas
  • BERT cannot generate text — encoder-only; for generation use GPT (decoder) or T5 (encoder-decoder)
  • attention_mask is mandatory when batching — padding tokens get mask=0 so attention ignores them
  • Truncation loses information — 512-token BERT limit discards the second half of long documents; chunk or use long-context model
  • Raw [CLS] ≠ good sentence embedding — use sentence-transformers (all-MiniLM-L6-v2) for retrieval/clustering
  • Word2Vec embeddings are static — "bank" gets the same vector regardless of context; polysemy is unsolvable
  • Tokenisation is model-specific — BERT uses WordPiece, GPT-2 uses BPE; token counts are not comparable
Hugging Face BERT Pattern
from transformers import AutoTokenizer, AutoModel
import torch

tokenizer = AutoTokenizer.from_pretrained('bert-base-uncased')
inputs = tokenizer(
    "Hello, world!",
    return_tensors='pt',
    padding=True,
    truncation=True,
    max_length=512
)
# inputs: {'input_ids', 'attention_mask', 'token_type_ids'}

model = AutoModel.from_pretrained('bert-base-uncased')
with torch.no_grad():
    outputs = model(**inputs)
# outputs.last_hidden_state: (batch, seq_len, 768)
# outputs.pooler_output:     (batch, 768)  — [CLS] representation
Sentence Embeddings (Correct Approach)
# Don't use raw BERT [CLS] for similarity — use sentence-transformers
from sentence_transformers import SentenceTransformer
from sklearn.metrics.pairwise import cosine_similarity

model = SentenceTransformer('all-MiniLM-L6-v2')
sentences = ["The cat sat on the mat", "A feline rested on the rug"]
embeddings = model.encode(sentences)   # (2, 384)

sim = cosine_similarity(embeddings[0:1], embeddings[1:2])
print(f"Similarity: {sim[0][0]:.3f}")  # should be high (~0.85+)
