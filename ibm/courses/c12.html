Fundamentals of AI Agents Using RAG and LangChain

Where LLMs stop being standalone models and start being components of systems. RAG solves the context window and stale knowledge problem. LangChain provides the plumbing. Agents add the ability to take actions. The most industry-relevant course in the programme.

RAG

The RAG Pipeline — Two Phases

RAG solves two fundamental LLM limitations: stale knowledge (training cutoff) and finite context (can't process whole document libraries in one prompt).

Indexing (offline, one-time)
Documents → Load → Split into chunks → Embed chunks → Store in vector DB
Retrieval (online, per query)
Query → Embed → Similarity search → Top-k chunks → Augmented prompt → LLM → Answer
RAG does NOT update model weights — it provides context at inference time only. The LLM's parameters are unchanged. This is the most important conceptual distinction in this course.
Chunking

Chunk Size — The Most Impactful RAG Parameter

Small chunks (~200 chars)Large chunks (~2000 chars)
Retrieval precision Higher — more targeted Lower — may include irrelevant content
Context completeness Lower — may miss surrounding context Higher — more context per result
Best for Fact lookup, specific questions Conceptual questions, summaries
DB size More chunks, more vectors Fewer chunks, fewer vectors

Chunk overlap ensures sentences split across chunk boundaries appear complete in at least one chunk. Rule of thumb: chunk_overlap ≥ 20% of chunk_size.

Agents

AI Agents — The ReAct Pattern

An agent uses an LLM as a reasoning engine to decide what actions to take in a Reason + Act loop:

Thought: I need to find the current population of France.
Action: search_tool("France population 2024")
Observation: France has approximately 68 million people.
Thought: I now have the answer.
Final Answer: France's population is approximately 68 million.
LangChain AbstractionRole
LLM / ChatModelThe reasoning engine — generates thoughts and action decisions
ToolA function the agent can call (search, calculator, code runner)
AgentExecutorRuns the Thought/Action/Observation loop; handles tool calls and errors
MemoryStores conversation history (ConversationBufferMemory, ConversationSummaryMemory)

Exam Traps
⚠ Trap 1 — RAG does not update weights
RAG provides external context at inference time — model weights are never changed. This is distinct from fine-tuning, where weights are updated. RAG makes the model's knowledge dynamic and up-to-date without the computational cost of fine-tuning.
⚠ Trap 2 — Embedding model must match
The embedding model used to index documents must be identical to the one used at retrieval. Different embedding models produce different vector spaces. Mixing them means your query vector and document vectors live in different geometric spaces — similarity scores are meaningless.
⚠ Trap 3 — FAISS is not persistent by default
FAISS stores the index in memory only. Call vectorstore.save_local('path/') after building to persist it. Otherwise the entire index is lost when the Python process ends and must be rebuilt from scratch.
⚠ Trap 4 — Agents can loop indefinitely
Agents are non-deterministic and can get stuck in loops or produce malformed tool calls. Always set max_iterations in AgentExecutor. Without it, a confused agent can call tools indefinitely, incurring API costs or hanging.
⚠ Trap 5 — ConversationBufferMemory grows unboundedly
ConversationBufferMemory does NOT automatically trim itself. Every message is stored verbatim. In a long conversation, this will eventually exceed the LLM's context window. Use ConversationSummaryMemory or manually truncate for long sessions.

Common Gotchas
  • Embedding model mismatch — index and query must use identical embedding model
  • FAISS not persistent — call vectorstore.save_local() after building; otherwise lost on process exit
  • LLM still hallucinates with RAG — RAG reduces hallucination but doesn't eliminate it; include explicit "answer only from context" instructions
  • Agents can loop indefinitely — always set max_iterations in AgentExecutor
  • ConversationBufferMemory grows unboundedly — use SummaryMemory for long sessions
  • LangChain API changes frequently — pin your version; 0.0.x, 0.1.x, 0.2.x are all different
  • k × chunk_size can overflow context — ensure retrieved tokens + prompt + query fits within model context window
RAG Pipeline Pattern
from langchain_community.document_loaders import PyPDFLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_community.embeddings import HuggingFaceEmbeddings
from langchain_community.vectorstores import FAISS

# Indexing
loader   = PyPDFLoader("document.pdf")
docs     = loader.load()
splitter = RecursiveCharacterTextSplitter(chunk_size=500, chunk_overlap=50)
chunks   = splitter.split_documents(docs)
embedder = HuggingFaceEmbeddings(model_name='all-MiniLM-L6-v2')
vs       = FAISS.from_documents(chunks, embedder)
vs.save_local('faiss_index')    # persist!

# Retrieval
vs = FAISS.load_local('faiss_index', embedder)  # same embedder!
results = vs.similarity_search("What is fine-tuning?", k=4)
LCEL RAG Chain
from langchain_openai import ChatOpenAI
from langchain_core.runnables import RunnablePassthrough
from langchain_core.output_parsers import StrOutputParser
from langchain.prompts import PromptTemplate

llm = ChatOpenAI(model='gpt-3.5-turbo', temperature=0)
prompt = PromptTemplate.from_template(
    "Context: {context}\nQuestion: {question}\nAnswer:"
)

chain = (
    {"context": vs.as_retriever(), "question": RunnablePassthrough()}
    | prompt
    | llm
    | StrOutputParser()
)
answer = chain.invoke("What is QLoRA?")
ReAct Agent Pattern
from langchain.agents import create_react_agent, AgentExecutor
from langchain.tools import Tool

tools = [
    Tool(name='Search', func=search_fn,
         description='Search the web for current information'),
]

agent = create_react_agent(llm, tools, prompt)
executor = AgentExecutor(
    agent=agent, tools=tools,
    max_iterations=10,          # prevent infinite loops
    verbose=True
)
executor.invoke({'input': 'What is the population of France?'})
