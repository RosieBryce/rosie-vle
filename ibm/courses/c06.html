AI Capstone Project with Deep Learning

A project course — less new theory, more applying Courses 1–5 to a real problem. Typically image classification using transfer learning. The marks are in the methodology: proper splitting, correct preprocessing, meaningful evaluation, and clear reporting.

Methodology

The Capstone Process — What Examiners Look For

This course tests process correctness, not model performance. The correct steps, in order:

1
Load & Explore — class distribution, sample visualisations, image dimensions and formats
2
Preprocess — resize to model's expected input, normalise with correct mean/std, augment training data only
3
Phase 1 — Feature Extraction — freeze base, train head only with large LR (1e-3), 5–10 epochs
4
Phase 2 — Fine-tuning — unfreeze top N layers, very small LR (1e-5), more epochs with early stopping
5
Evaluate on Test Set — accuracy, precision, recall, F1, confusion matrix, error analysis
6
Report & Compare — compare against a baseline; conclusions; what you'd try next
The test set must be completely held out. Evaluating on validation inflates performance — you've implicitly optimised for val through early stopping and hyperparameter choices. Only the test set is an honest estimate of real-world performance.
Architecture

Choosing a Pretrained Backbone

ModelSizeSpeedWhen to use
MobileNetV2/V3 Small Fast Mobile, edge, small datasets, prototype
ResNet-50 Medium Medium General purpose — safe default
EfficientNetB0–B7 Varies Medium When accuracy per parameter is priority
VGG16 Large Slow Older benchmark; avoid for new projects
ViT (Vision Transformer) Large Slow SOTA quality when data is abundant
Transfer learning works best when your dataset is small and the source domain (ImageNet) is similar to your target domain. Feature extraction (frozen) trains faster; fine-tuning achieves higher accuracy when you have enough data.
Evaluation

Required Evaluation Components

ComponentPurpose / What to look for
Training curves (loss + accuracy)Divergence between train/val = overfitting; flat curves = underfitting or LR issue
Classification reportPer-class precision, recall, F1 — not just overall accuracy
Confusion matrixWhich classes are confused with which — reveals systematic errors
Error analysisDisplay wrong predictions — are confused classes visually similar?
Baseline comparisonCompare against random, logistic regression, or shallow CNN from scratch

If classes are imbalanced, report per-class F1 and recall, not just accuracy. Consider class_weight in Keras or weighted sampling in PyTorch.

Exam Traps
⚠ Trap 1 — Evaluating on val, not test
Reporting validation set performance as the final result is a methodological error. The val set was used during training (early stopping monitors it; hyperparameters are tuned against it). It is not held-out — only the test set, never seen during any training decision, is the honest estimate.
⚠ Trap 2 — Preprocessing mismatch
Each pretrained model requires its own specific preprocessing function. VGG16 expects BGR channels with ImageNet mean subtracted. MobileNetV2 expects [-1, 1]. Using the wrong preprocessing produces no error — images are valid tensors — but accuracy silently degrades. Always use the model's accompanying preprocess_input() function.
⚠ Trap 3 — Fine-tuning LR must be much smaller
Fine-tuning learning rate should be ~10–100× smaller than training from scratch. A high LR during fine-tuning destroys the pretrained feature representations (catastrophic forgetting). The typical pattern: 1e-3 for head training, 1e-5 for fine-tuning.
⚠ Trap 4 — Augmentation on training data only
Data augmentation must be applied to training data only. Val and test sets should receive only standard preprocessing (resize + normalise). Augmenting val/test introduces randomness into your evaluation metrics.
⚠ Trap 5 — No baseline = incomplete project
A project without a baseline comparison cannot demonstrate whether transfer learning helped. At minimum, compare against logistic regression on flattened pixel values or a shallow CNN trained from scratch. Without a reference point, a 90% accuracy number is meaningless.

Common Gotchas
  • Evaluating on val instead of test — test set must be completely held out until final evaluation
  • Preprocessing mismatch — each pretrained model has its own preprocess_input; using wrong one silently degrades accuracy
  • Augmenting val/test — augmentation must be training-only; evaluation should be deterministic
  • shuffle=False in test DataLoader — if you shuffle test set, y_pred and y_true arrays won't align
  • No baseline — always compare against something simpler to show transfer learning actually helps
  • Class imbalance ignored — report F1/recall per class, not just accuracy
Two-Phase Transfer Learning (Keras)
# Phase 1: Feature extraction
base_model.trainable = False
optimizer = Adam(lr=1e-3)
model.compile(optimizer=optimizer, loss='categorical_crossentropy', metrics=['accuracy'])
model.fit(train_ds, epochs=5, validation_data=val_ds)

# Phase 2: Fine-tuning
base_model.trainable = True
for layer in base_model.layers[:-30]:    # freeze all but top 30
    layer.trainable = False

optimizer = Adam(lr=1e-5)               # MUCH smaller LR
model.compile(optimizer=optimizer, loss='categorical_crossentropy', metrics=['accuracy'])
model.fit(train_ds, epochs=20,
          validation_data=val_ds,
          callbacks=[EarlyStopping(patience=5, restore_best_weights=True)])
Evaluation Pattern
from sklearn.metrics import classification_report, confusion_matrix
import seaborn as sns

y_pred = np.argmax(model.predict(test_ds), axis=1)
y_true = np.concatenate([y for _, y in test_ds])

print(classification_report(y_true, y_pred, target_names=class_names))

cm = confusion_matrix(y_true, y_pred)
sns.heatmap(cm, annot=True, fmt='d',
            xticklabels=class_names,
            yticklabels=class_names)

# Error analysis
errors = np.where(y_pred != y_true)[0]
for idx in errors[:5]:
    print(f"True: {class_names[y_true[idx]]} | Pred: {class_names[y_pred[idx]]}")
