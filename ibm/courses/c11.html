Generative AI Advanced Fine-Tuning for LLMs

The hardest conceptual course in the Gen AI block. RLHF, DPO, QLoRA, and quantisation. You don't need to implement RLHF from scratch, but you need to understand the pipeline, why it exists, and its failure modes. QLoRA is the practical technique you will use.

📖 Flipped Classroom — Before You Watch
Prime your thinking — bring these questions into the lectures
  • 1 RLHF uses a reward model and PPO to align an LLM's outputs with human preferences. What failure mode do you predict if the model is allowed to train against the reward model for too long without any constraint? What mechanism prevents this?
  • 2 DPO claims to achieve alignment without a reward model or RL. How can it be possible to train on preference data (which response is better) without explicitly modelling a reward? What is being implicitly derived?
  • 3 Storing a 7B parameter model in fp32 requires ~28GB. In 4-bit (NF4) it requires ~3.5GB. What is being lost when you reduce precision, and why does NF4 lose less quality than naive int4 for transformer weights specifically?
Watch for…
The KL divergence penalty in RLHF — it's not just a technical detail, it's the mechanism preventing reward hacking. Also: DPO's key insight that the optimal RL policy has a closed-form solution derivable from the SFT reference model, making a reward model unnecessary. And: NF4 vs int4 — normal float 4-bit is optimised for normally distributed values, which is why it fits transformer weights better than naive int4.
Core Theory Assessment priority — RLHF pipeline stages, DPO vs RLHF, and QLoRA mechanics
RLHF

RLHF — Three Stages

Stage 1
SFT
Fine-tune base model on high-quality instruction-response pairs. Creates instruction-following SFT model.
Stage 2
Reward Model
Humans compare response pairs; a separate model learns to predict human preference as a scalar reward score. This stage IS supervised learning.
Stage 3
PPO
RL updates LLM to maximise reward score, constrained by KL divergence from SFT model to prevent reward hacking.
The KL penalty is critical: Reward = R(response) − β × KL(π ∥ π_SFT). Without it, the model quickly finds adversarial patterns that fool the reward model without being genuinely better (reward hacking). The KL term anchors the model to the SFT baseline.
Alignment Failures

RLHF Failure Modes

ProblemDescriptionMitigation
Reward hacking Model exploits reward model weaknesses; gets high reward without being genuinely better KL penalty, ensemble reward models
Mode collapse Model converges to one type of safe, repetitive response Diversity rewards, sampling during RL
Sycophancy Model tells users what they want to hear even when wrong Adversarial preference data
Overoptimisation Too many PPO steps degrades general capabilities Early stopping, tight KL constraint
DPO

DPO — Direct Preference Optimisation

DPO achieves alignment without a reward model or RL. Key insight: the optimal RL policy has a closed-form solution expressible directly in terms of the reference SFT model. Train directly on preference data.

RLHFDPO
Reward modelRequired (separate model)Not needed
Training algorithmPPO (RL)Standard gradient descent
StabilityComplex, sensitiveMore stable
Reference modelKL constraint from π_SFTFrozen π_ref required
AdoptionOpenAI, early ChatGPTNow the default in open-source
DPO requires a frozen reference model (π_ref). Without it, there is no calibration point and training diverges. The reference model must be the same SFT model you're starting from, kept frozen throughout DPO training.
QLoRA

Quantisation & QLoRA

PrecisionBits7B model sizeQuality
fp3232~28 GBPerfect reference
fp16 / bf1616~14 GBEssentially identical

NF4 (QLoRA)4~3.5 GBUsable; better than naive int4

QLoRA combines 4-bit quantisation with LoRA: load model weights in 4-bit (NF4), keep frozen, train only LoRA adapters in bf16. Fine-tune a 7B model on a 16GB consumer GPU.

QLoRA does NOT fine-tune the quantised weights — only the LoRA adapters are updated (in higher precision). The base model weights stay fixed in 4-bit throughout training. NF4 is better than naive int4 because transformer weights follow roughly normal distributions — NF4 is specifically designed for normally distributed values.

Exam Traps
⚠ Trap 1 — RLHF Stage 2 is supervised, not RL
Training the reward model (Stage 2) is supervised learning, not reinforcement learning. Humans provide pairwise preference labels — the reward model is trained via standard gradient descent to predict which response is better. RL only appears in Stage 3 (PPO).
⚠ Trap 2 — PPO is the RL algorithm, not Q-learning
RLHF uses Proximal Policy Optimisation (PPO). Common wrong answers: Q-learning, A3C, DQN. PPO is chosen because it's stable, on-policy, and the KL constraint integrates naturally. PPO is NOT the same as DPO.
⚠ Trap 3 — DPO does NOT require a reward model
DPO's defining advantage over RLHF is that it requires no separate reward model. It trains directly on (prompt, chosen_response, rejected_response) triples using standard gradient descent. However, it still requires a frozen reference model (the SFT model) for calibration.
⚠ Trap 4 — QLoRA does not update quantised weights
In QLoRA, the base model weights remain frozen in 4-bit throughout training. Only the LoRA adapter matrices are updated — and they are kept in higher precision (bf16/fp16). The quantised weights are never directly modified. This is the "Quantised" in QLoRA.
⚠ Trap 5 — NF4 vs int4 quality difference
NF4 is better than naive int4 for transformer weights because transformer weights roughly follow a normal distribution. NF4 places quantisation levels at positions that minimise error specifically for normally distributed values. Int4 places them at uniform intervals, which is suboptimal for values concentrated near zero.

Common Gotchas
  • DPO needs frozen reference model — ref_model must be the same SFT model, frozen throughout; without it training diverges
  • 4-bit quantisation requires bitsandbytes — currently NVIDIA CUDA only; no MPS (Mac) or CPU training support
  • device_map='auto' incompatible with manual .to(device) — don't combine them; auto handles device placement
  • KL penalty β in DPO — critical hyperparameter; too small = model diverges; too large = fine-tuning has no effect; range 0.01–0.5
  • Merge LoRA before deployment — model.merge_and_unload() eliminates inference overhead
  • Reward hacking is real — monitor both reward score AND human evaluation; reward score alone is insufficient
QLoRA Pattern
from transformers import BitsAndBytesConfig, AutoModelForCausalLM
from peft import LoraConfig, get_peft_model
import torch

bnb_config = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_quant_type='nf4',           # NF4 > int4 for normal distributions
    bnb_4bit_use_double_quant=True,      # quantise quantisation constants too
    bnb_4bit_compute_dtype=torch.bfloat16
)

model = AutoModelForCausalLM.from_pretrained(
    'meta-llama/Llama-2-7b-hf',
    quantization_config=bnb_config,
    device_map='auto'                    # distribute across available devices
)

lora_config = LoraConfig(r=64, lora_alpha=16, target_modules=['q_proj','v_proj'])
model = get_peft_model(model, lora_config)
# Only LoRA adapters train; base weights stay frozen in 4-bit
